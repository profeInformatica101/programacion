01 — Fundamentos

Persistencia & JPA

Jakarta Persistence API (JPA) es la especificación estándar de Java para la gestión de persistencia y el mapeo objeto-relacional (ORM). Permite que las aplicaciones guarden y recuperen objetos Java en una base de datos relacional sin escribir SQL manual, usando en su lugar anotaciones declarativas.

La persistencia consiste en garantizar que el estado de un objeto sobrevive más allá del tiempo de ejecución de la aplicación. JPA proporciona una capa de abstracción sobre JDBC que normaliza el acceso a datos entre distintos proveedores.

Abstracción ORM

Elimina el SQL repetitivo mediante anotaciones que mapean clases a tablas y atributos a columnas automáticamente.

Ciclo de vida gestionado

JPA controla los estados de las entidades: NEW, MANAGED, DETACHED y REMOVED.

Portabilidad

La misma API funciona con Hibernate, EclipseLink u OpenJPA. Cambiar de proveedor no requiere reescribir la lógica de negocio.

JPQL orientado a objetos

Un lenguaje de consultas que trabaja sobre entidades Java, no sobre tablas, garantizando independencia del motor de BD.

Alternativas a la persistencia en ficheros: Antes de JPA, los datos se guardaban en ficheros planos (serialización Java, XML, JSON) o con acceso JDBC manual. JPA unifica estas alternativas con un modelo declarativo estándar que gestiona automáticamente transacciones, caché y relaciones entre entidades.

02 — Entidades

Entidades JPA

Una entidad es una clase Java ligera (POJO) que se mapea directamente a una tabla de base de datos. Debe estar anotada con @Entity, poseer un constructor sin argumentos, y declarar al menos un campo @Id que actúe como clave primaria.

Java · Empleado.java 
import jakarta.persistence.*;

@Entity                                  // Clase mapeada a tabla
@Table(name = "empleados")              // Nombre explícito de la tabla
public class Empleado {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;                       // Clave primaria autogenerada

    @Column(name = "nombre_completo", nullable = false, length = 100)
    private String nombre;

    @Column(unique = true)
    private String email;

    @Basic(fetch = FetchType.LAZY, optional = true)
    private String fotoPerfil;             // Carga diferida

    @Transient
    private String campoNoMapeado;         // No se persiste en BD

    // Constructor sin argumentos obligatorio
    public Empleado() {}

    // Getters y setters...
}
@Entity @Table @Id @GeneratedValue @Column @Basic @Transient @Embeddable @Embedded

Estrategias de GenerationType: IDENTITY (autoincrement de BD), SEQUENCE (secuencia de BD), TABLE (tabla de secuencias JPA), AUTO (el proveedor elige la mejor opción).

03 — Configuración

Unidad de persistencia

El archivo META-INF/persistence.xml define la unidad de persistencia: el conjunto de entidades gestionadas, el proveedor JPA, el tipo de transacción y las propiedades de conexión a la base de datos.

XML · META-INF/persistence.xml 
<persistence version="3.0"
             xmlns="https://jakarta.ee/xml/ns/persistence">

  <persistence-unit name="miUnidad"
                    transaction-type="RESOURCE_LOCAL">

    <provider>org.hibernate.jpa.HibernatePersistenceProvider</provider>

    <!-- Clases entidad registradas -->
    <class>com.example.Empleado</class>
    <class>com.example.Departamento</class>

    <properties>
      <property name="jakarta.persistence.jdbc.url"
               value="jdbc:mysql://localhost:3306/empresa"/>
      <property name="jakarta.persistence.jdbc.user"     value="root"/>
      <property name="jakarta.persistence.jdbc.password" value="****"/>
      <property name="jakarta.persistence.schema-generation.database.action"
               value="create"/>
      <!-- Mostrar SQL generado en consola -->
      <property name="hibernate.show_sql" value="true"/>
    </properties>

  </persistence-unit>
</persistence>

RESOURCE_LOCAL

Transacciones gestionadas por la aplicación (Java SE). El desarrollador controla explícitamente begin(), commit() y rollback().

JTA

Transacciones gestionadas por el contenedor Jakarta EE (WildFly, GlassFish). Permite transacciones distribuidas entre varios recursos.

04 — EntityManager

Gestor de entidades

El EntityManager es la interfaz central de JPA. Gestiona el contexto de persistencia y el ciclo de vida completo de las entidades. En modo RESOURCE_LOCAL, las transacciones se controlan mediante EntityTransaction.

Java · Patrón de uso EntityManager 
// 1. Obtener el EntityManagerFactory (una sola vez por aplicación)
EntityManagerFactory emf =
    Persistence.createEntityManagerFactory("miUnidad");

// 2. Crear EntityManager (uno por operación / hilo)
EntityManager em = emf.createEntityManager();
EntityTransaction tx = em.getTransaction();

try {
    tx.begin();                       // Iniciar transacción

    Empleado emp = new Empleado();
    emp.setNombre("Laura García");
    emp.setEmail("laura@empresa.com");

    em.persist(emp);                  // NEW → MANAGED
    tx.commit();                      // Confirmar: INSERT en BD

} catch (Exception e) {
    if (tx.isActive()) tx.rollback(); // Deshacer cambios
    throw e;
} finally {
    em.close();                       // Liberar recursos
}

// 3. Cerrar factory al terminar la aplicación
emf.close();

Estados de las entidades

NEW — objeto creado, no gestionado.
MANAGED — dentro del contexto de persistencia.
DETACHED — fuera del contexto tras close().
REMOVED — marcado para borrar en la BD.

Métodos clave

persist(e) — NEW → MANAGED
merge(e) — DETACHED → MANAGED
remove(e) — MANAGED → REMOVED
find(C, id) — busca por PK
flush() — sincroniza con BD
refresh(e) — recarga desde BD

05 — Operaciones

Operaciones CRUD

JPA simplifica las cuatro operaciones fundamentales de acceso a datos. Las entidades en estado MANAGED propagan sus cambios automáticamente a la BD al hacer commit() de la transacción.

Create em.persist(e) Inserta una nueva entidad en la BD. El objeto pasa de NEW a MANAGED.
Read em.find(C, id) Busca por clave primaria. Devuelve null si no existe. Usa getReference() para carga perezosa.
Update em.merge(e) Fusiona una entidad desacoplada. Devuelve la instancia gestionada con los cambios aplicados.
Delete em.remove(e) Marca la entidad como REMOVED. El DELETE se ejecuta al commit. La entidad debe estar MANAGED.
Java · Ejemplo completo CRUD 
// CREATE
tx.begin();
Empleado nuevo = new Empleado();
nuevo.setNombre("Marcos");
em.persist(nuevo);
tx.commit();

// READ
Empleado e = em.find(Empleado.class, 1L);

// UPDATE
tx.begin();
e.setNombre("Marcos Ruiz");       // dirty checking automático
tx.commit();

// DELETE
tx.begin();
Empleado managed = em.find(Empleado.class, 1L);
em.remove(managed);
tx.commit();

Dirty Checking: Si una entidad está en estado MANAGED y modificas sus atributos dentro de una transacción, JPA detecta automáticamente los cambios y genera el UPDATE correspondiente sin necesidad de llamar a ningún método adicional.

06 — Consultas

JPQL — Java Persistence Query Language

JPQL es un lenguaje de consultas orientado a objetos inspirado en SQL, pero que opera sobre entidades Java y sus atributos en lugar de sobre tablas y columnas. Es portable entre proveedores y produce SQL nativo de forma transparente.

Consulta SELECT con parámetros
Java · TypedQuery 
// Consulta tipada (recomendada sobre Query)
String jpql = "SELECT e FROM Empleado e WHERE e.departamento.nombre = :dept";
TypedQuery<Empleado> q = em.createQuery(jpql, Empleado.class);
q.setParameter("dept", "Ingeniería");
q.setMaxResults(10);                   // Paginación
q.setFirstResult(0);
List<Empleado> lista = q.getResultList();

// Consulta de resultado único
Empleado uno = em.createQuery(
    "SELECT e FROM Empleado e WHERE e.email = :mail", Empleado.class)
    .setParameter("mail", "admin@empresa.com")
    .getSingleResult();

// Actualización masiva
int actualizados = em.createQuery(
    "UPDATE Empleado e SET e.sueldo = e.sueldo * 1.1 WHERE e.antiguedad > 5")
    .executeUpdate();

// Eliminación masiva
em.createQuery("DELETE FROM Empleado e WHERE e.activo = false").executeUpdate();

// Proyección con constructor
List<EmpleadoDTO> dtos = em.createQuery(
    "SELECT NEW com.example.EmpleadoDTO(e.nombre, e.email) FROM Empleado e",
    EmpleadoDTO.class).getResultList();

Funciones JPQL

Agregados: COUNT(), SUM(), AVG(), MAX(), MIN()
Cadenas: UPPER(), LOWER(), CONCAT(), LENGTH()
Fechas: CURRENT_DATE, CURRENT_TIMESTAMP

Named Queries

Consultas predefinidas anotadas directamente en la entidad con @NamedQuery. Se compilan en el arranque, mejorando el rendimiento y detectando errores temprano.

Criteria API

API programática type-safe como alternativa a JPQL en cadena de texto. Ideal para consultas dinámicas construidas en tiempo de ejecución sin concatenación de strings.

07 — Herencia

Herencia en JPA

JPA permite mapear jerarquías de clases Java a tablas relacionales mediante la anotación @Inheritance. La elección de estrategia afecta directamente al rendimiento, la normalización y la complejidad de las consultas.

SINGLE_TABLE

Toda la jerarquía en una única tabla. Usa una @DiscriminatorColumn para identificar el tipo. Estrategia por defecto: máximo rendimiento, consultas sin JOINs, pero con columnas nulas.

JOINED

Una tabla por clase, unidas por la clave primaria. Diseño más normalizado. Recomendada para jerarquías con muchos atributos específicos por subclase y consultas polimórficas.

TABLE_PER_CLASS

Una tabla completa por clase concreta con todos sus atributos. No permite columnas NULL, pero dificulta las consultas polimórficas al requerir UNIONs. No siempre soportada obligatoriamente.

Java · Herencia SINGLE_TABLE 
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorColumn(name = "tipo", discriminatorType = DiscriminatorType.STRING)
public abstract class Persona {
    @Id @GeneratedValue private Long id;
    private String nombre;
}

@Entity
@DiscriminatorValue("EMP")
public class Empleado extends Persona {
    private String puesto;
}

@Entity
@DiscriminatorValue("CLI")
public class Cliente extends Persona {
    private String cif;
}
@Inheritance InheritanceType @DiscriminatorColumn @DiscriminatorValue @MappedSuperclass
08 — Relaciones

Asociaciones y cardinalidad

Las relaciones entre entidades reflejan las claves foráneas del modelo relacional. JPA las expresa mediante anotaciones que especifican la cardinalidad (cuántos) y la navegabilidad (en qué dirección).

Java · Bidireccional @OneToMany / @ManyToOne 
@Entity
public class Departamento {
    @Id private Long id;
    private String nombre;

    @OneToMany(mappedBy = "departamento",
               cascade = CascadeType.ALL,
               orphanRemoval = true)
    private List<Empleado> empleados = new ArrayList<>();
}

@Entity
public class Empleado {
    @Id private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "depto_id")       // FK en tabla empleados
    private Departamento departamento;
}

// ManyToMany con tabla de unión personalizada
@ManyToMany
@JoinTable(name = "empleado_proyecto",
    joinColumns = @JoinColumn(name = "emp_id"),
    inverseJoinColumns = @JoinColumn(name = "proy_id"))
private List<Proyecto> proyectos;

Cardinalidades disponibles

@OneToOne — Un objeto ↔ Un objeto
@OneToMany — Un objeto → Muchos
@ManyToOne — Muchos → Un objeto
@ManyToMany — Muchos ↔ Muchos (tabla intermedia)

Cascade & Fetch

cascade: propaga operaciones (PERSIST, MERGE, REMOVE, ALL) a entidades relacionadas.

fetch: EAGER (carga inmediata) o LAZY (carga diferida al acceder). @ManyToOne es EAGER por defecto, @OneToMany es LAZY.

mappedBy y @JoinColumn

mappedBy indica el lado inverso en una relación bidireccional (la que no tiene la clave foránea). @JoinColumn especifica el nombre de la columna FK en la base de datos.

09 — Ingeniería inversa

Creación de entidades desde BD

La ingeniería inversa (reverse engineering) permite generar automáticamente las clases @Entity y el persistence.xml a partir de un esquema relacional existente, agilizando significativamente el desarrollo.

NetBeans

Entidades desde base de datos

Asistente integrado: New → Entity Classes from Database. Conecta a la BD mediante un DataSource o driver JDBC, selecciona las tablas y genera automáticamente las entidades con sus relaciones, el persistence.xml y el controlador JPA (Facade pattern).

IntelliJ IDEA

JPA Reverse Engineering

Herramienta Persistence → Generate Persistence Mapping → By Database Schema. Genera entidades con soporte para estrategias de herencia, tipos de fetch y relaciones bidireccionales. Incluye editor visual de mapeos ORM y validación JPQL en tiempo real.

Eclipse Dali

JPA Tools — Generate Entities

Plugin Dali incluido en Eclipse IDE for Enterprise. Genera entidades desde tablas con asistente paso a paso y editor gráfico del persistence.xml. Soporta asociaciones complejas y personalización de columnas.

Hibernate Tools

hbm2java / reverse engineering

Plugin Maven / Gradle que genera entidades JPA, archivos de mapeo HBM y esquemas DDL a partir de una BD existente. Configurable mediante fichero reveng.xml.

Flujo típico de ingeniería inversa: (1) Crear conexión JDBC al SGBD en el IDE → (2) Seleccionar esquema y tablas → (3) Configurar opciones (nombre de paquete, estrategia de herencia) → (4) Generar entidades, persistence.xml y controladores JPA → (5) Revisar y ajustar las anotaciones generadas.

10 — Ecosistema

Controladores JPA & Herramientas

Los controladores de entidad (Entity Facades o Stateless Session Beans) encapsulan las operaciones CRUD en clases de servicio reutilizables. Los IDEs modernos los generan automáticamente siguiendo el patrón DAO / Repository.

Java · Controlador JPA (patrón Facade) 
public class EmpleadoFacade {

    private final EntityManagerFactory emf;

    public EmpleadoFacade(EntityManagerFactory emf) {
        this.emf = emf;
    }

    public void create(Empleado e) {
        EntityManager em = emf.createEntityManager();
        try {
            em.getTransaction().begin();
            em.persist(e);
            em.getTransaction().commit();
        } finally { em.close(); }
    }

    public Empleado findById(Long id) {
        EntityManager em = emf.createEntityManager();
        try {
            return em.find(Empleado.class, id);
        } finally { em.close(); }
    }

    public List<Empleado> findAll() {
        EntityManager em = emf.createEntityManager();
        try {
            return em.createQuery("SELECT e FROM Empleado e", Empleado.class)
                     .getResultList();
        } finally { em.close(); }
    }

    public void remove(Long id) {
        EntityManager em = emf.createEntityManager();
        try {
            em.getTransaction().begin();
            Empleado ref = em.find(Empleado.class, id);
            if (ref != null) em.remove(ref);
            em.getTransaction().commit();
        } finally { em.close(); }
    }
}

Gestión persistence.xml

NetBeans e IntelliJ IDEA ofrecen editores visuales para el archivo persistence.xml: agregar/eliminar entidades, configurar conexiones y ajustar propiedades del proveedor sin editar XML manualmente.

Asistentes de conexión

Los IDEs permiten crear y gestionar fuentes de datos (DataSources) para MySQL, PostgreSQL, H2, Oracle, etc. con asistentes visuales que configuran drivers y cadenas de conexión JDBC.

Apache Maven / Gradle

Dependencia estándar: jakarta.persistence:jakarta.persistence-api:3.1.0. Hibernate Validator y plugins de generación de esquemas DDL simplifican el ciclo de build completo.

Buenas prácticas: Usar RESOURCE_LOCAL con patrón Facade en Java SE; preferir JTA con @Stateless en Jakarta EE. Definir consultas como @NamedQuery en las entidades. Activar hibernate.show_sql solo en desarrollo. Cerrar siempre el EntityManager en un bloque finally.
